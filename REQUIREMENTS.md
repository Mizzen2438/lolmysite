# 要件定義書 — ゲーム仲間募集サービス(仮称: LoL MySite)

- 版数: 1.1(ドラフト)— 複数アカウント/スマーフ禁止と Riot API ランク連携をフェーズ 1 に追加
- 作成日: 2026-06-12
- 対象フェーズ: フェーズ 1(League of Legends 専門の MVP)

---

## 1. サービス概要・背景

### 1.1 背景と課題

現在、ゲームを一緒にプレイするメンバーの募集・応募は、X(Twitter)のハッシュタグ投稿や Discord の募集チャンネルなど、汎用 SNS 上で散発的に行われている。これには次の課題がある。

| 課題 | 内容 |
|---|---|
| 募集が流れる | タイムライン型の SNS では募集投稿がすぐに埋もれ、必要な人に届かない |
| 条件が伝わらない | ランク帯・レーン・VC 有無・開始時刻などの条件が自由文で書かれ、認識齟齬が起きる |
| 検索できない | 「今夜 22 時から、ゴールド帯、サポート募集」のような条件検索ができない |
| 応募管理が煩雑 | リプライや DM が交錯し、誰を承認したか・定員が埋まったかの管理が手作業になる |
| 集合導線が長い | 成立後に Discord サーバー招待や VC の案内を個別に DM で送る必要がある |
| 安全性の担保がない | 相手の評判が分からず、無断キャンセルや暴言などのトラブルを避けられない |

### 1.2 サービスの価値(コンセプト)

**「募集を立てる → 条件で見つかる → 応募を承認する → そのまま Discord に集まる」までを一気通貫で完結させる、ゲーム仲間募集の専門プラットフォーム。**

- 募集を構造化データとして扱い、条件での検索・フィルタを可能にする
- 応募・承認・定員管理をシステム化し、募集主の手間を最小化する
- Discord 連携を中核に据え、成立後すぐに集合できる導線を提供する
- 評価・通報の仕組みでコミュニティの健全性を保つ

### 1.3 名称・対象ゲーム

- フェーズ 1 では **League of Legends(LoL)専門**として開始する
- データ設計上はゲームをマスタデータとして持ち、将来 Valorant・Apex Legends・モンスターハンター等へ拡張可能にする(→ 8 章)

## 2. ターゲットユーザー

### 2.1 ユーザー像

- 日本国内の LoL プレイヤー(PC)
- 野良(ソロキュー)に疲れ、固定または単発のフルパーティ/デュオ相手を探している層
- すでに X / Discord で「#LoL募集」等を使って募集・応募の経験がある層

### 2.2 ペルソナ

| | ペルソナ A | ペルソナ B | ペルソナ C |
|---|---|---|---|
| 名前 | ソロランナー(24歳・社会人) | エンジョイ勢(19歳・学生) | 主催者気質(28歳・社会人) |
| ランク | ゴールド | アンランク〜ブロンズ | プラチナ |
| 動機 | デュオ相手を探してランクを上げたい | ノーマル/ARAM をワイワイ遊びたい。VC は聞き専希望 | クラン的な固定メンバーを集めて毎週活動したい |
| 課題 | X の募集はランク帯が合わない相手からの応募が多い | 雰囲気が怖い募集を避けたい | 応募者の管理と毎回の集合案内が面倒 |
| 主な利用機能 | 条件フィルタ検索、デュオ募集 | 雰囲気タグ(エンジョイ/初心者歓迎)、聞き専 OK フィルタ | 定期募集、応募承認管理、Discord サーバー招待自動提示 |

## 3. スコープ

### 3.1 フェーズ定義

| フェーズ | 内容 |
|---|---|
| フェーズ 1(本書の対象 / MVP) | LoL 専門。募集の作成・検索・応募・承認・Discord 集合導線・基本的な通知と安全機能。Riot API 連携によるランク自動取得と、アカウント一意性(サブ垢・スマーフ禁止)の担保 |
| フェーズ 2 | 評価(GG レビュー)の本格運用、定期/固定メンバー募集、Discord Bot による通知強化、SMS 電話番号認証の導入判断(F-UNIQ-09)、Riot Sign On による本人所有確認の導入検討(F-ACC-09) |
| フェーズ 3 | 他ゲームへの拡張(ゲームマスタの追加)、スマホアプリ化の検討 |

### 3.2 フェーズ 1 で対象外とするもの

- 他ゲーム対応(設計のみ拡張可能にしておく)
- サイト内リアルタイムチャット(連絡は Discord に寄せる)
- 大会・スクリム運営機能
- 有料プラン・課金機能

## 4. 機能要件

要件 ID は `F-カテゴリ-連番`。優先度: **Must**(MVP 必須)/ **Should**(MVP に入れたい)/ **Could**(フェーズ 2 以降)。

### 4.1 アカウント・プロフィール

| ID | 要件 | 優先度 |
|---|---|---|
| F-ACC-01 | ユーザーは Discord OAuth2 でログイン/新規登録できる(独自パスワードは持たない) | Must |
| F-ACC-02 | 初回ログイン時、Discord のユーザー名・アバターをプロフィールに自動取り込みする | Must |
| F-ACC-03 | プロフィールに Riot ID(ゲーム名#タグライン)を登録できる。登録時に Riot API(Account-V1)で実在確認を行う | Must |
| F-ACC-04 | プロフィールに得意レーン(複数可)、主なプレイ時間帯、自己紹介文を登録できる | Must |
| F-ACC-05 | プロフィールに VC スタイル(通話 OK / 聞き専 / テキストのみ)を登録できる | Must |
| F-ACC-06 | Riot API(League-V4)と連携し、ランク(ソロ/フレックス)を自動取得してプロフィールに表示する。ランクは自己申告できず、API 取得値のみを正とする(スマーフ申告の防止) | Must |
| F-ACC-07 | ユーザーは退会でき、退会時に個人情報は削除される(募集・応募履歴は匿名化して保持) | Must |
| F-ACC-08 | ランクは定期的(例: 24 時間ごと)およびユーザーの手動更新操作で再取得し、最終取得日時を表示する | Must |
| F-ACC-09 | Riot Sign On(RSO)による Riot アカウントの本人所有確認。MVP では PUUID の一意登録(F-UNIQ-03)+通報運用でなりすましに対応し、フェーズ 2 で RSO 導入を検討する | Could |

#### アカウントの一意性(サブ垢・スマーフ対策)

| ID | 要件 | 優先度 |
|---|---|---|
| F-UNIQ-01 | アカウントは 1 人 1 つとする。複数アカウント(サブ垢)の作成・利用を利用規約で禁止する | Must |
| F-UNIQ-02 | 1 つの Discord アカウントにつき登録できるサービスアカウントは 1 つのみとする(Discord ID で一意化) | Must |
| F-UNIQ-03 | 1 つの Riot アカウント(PUUID)を登録できるユーザーは 1 人のみとする。他ユーザーが登録済みの Riot ID は登録できない。なお本人所有の確認(他人の Riot ID の登録)までは保証しないため、なりすましは通報(F-SAFE-01)と RSO 導入検討(F-ACC-09)で対応する | Must |
| F-UNIQ-04 | 退会後の再登録時、同一 Discord ID の過去の凍結・警告履歴を引き継ぐ(凍結逃れのための作り直しを防止) | Must |
| F-UNIQ-05 | スマーフ行為(メインより低ランクのアカウントでの参加、実力を偽った低ランク帯募集への参加)を利用規約・ガイドラインで禁止する | Must |
| F-UNIQ-06 | 運営者は同一人物による複数アカウントの疑いを調査し、該当アカウントを凍結できる | Must |
| F-UNIQ-07 | 登録時に Discord アカウントの作成日時(Discord ID の Snowflake から算出)を確認し、作成から一定期間(例: 3 ヶ月)未満のアカウントは登録できない(作りたてのサブ垢の抑止) | Must |
| F-UNIQ-08 | 同一 IP アドレス・デバイスフィンガープリントからの複数アカウント利用を検知し、運営者の管理画面に重複疑いとして表示する(家族・共有回線の誤検知があり得るため自動凍結はせず、F-UNIQ-06 の調査・凍結の判断材料とする) | Should |
| F-UNIQ-09 | SMS による電話番号認証(1 電話番号 = 1 アカウント)。スマーフ・サブ垢被害が実際に問題化した場合の追加対策としてフェーズ 2 で導入を判断する | Could |

> 注: 本セクションの要件は同一人物の複数登録を完全に防ぐものではなく、「アカウントの使い回し・量産のコストを上げる(抑止)」+「通報・調査により検知して凍結する」ことを目的とする。完全な防止には eKYC 等の本人確認が必要になるが、登録時の離脱が大きく本サービスでは採用しない。

### 4.2 募集(Recruitment)

| ID | 要件 | 優先度 |
|---|---|---|
| F-REC-01 | ログインユーザーは募集を作成できる | Must |
| F-REC-02 | 募集には次の構造化項目を設定する: ゲームモード(ランク/ノーマル/ARAM/カスタム等)、募集レーンと人数、対象ランク帯(下限〜上限)、開始予定時刻、想定プレイ時間、VC 有無と使用ツール、雰囲気タグ、フリーコメント | Must |
| F-REC-03 | 雰囲気タグは定義済みの選択式とする(例: ガチ、エンジョイ、初心者歓迎、聞き専 OK、社会人、学生、深夜帯) | Must |
| F-REC-04 | 募集主は募集を編集・締切・削除できる | Must |
| F-REC-05 | 開始予定時刻を過ぎた募集は自動で期限切れ(クローズ)になる | Must |
| F-REC-06 | 定員が埋まった募集は自動で「成立」状態になる | Must |
| F-REC-07 | 募集ステータスは「募集中 → 成立 / 締切 / 期限切れ」のライフサイクルで管理する | Must |
| F-REC-08 | 毎週同じ曜日・時刻の定期募集を作成できる | Could |
| F-REC-09 | 募集をテンプレートとして保存し再利用できる | Should |

### 4.3 検索・一覧

| ID | 要件 | 優先度 |
|---|---|---|
| F-SRCH-01 | 募集一覧を新着順・開始時刻順で表示できる | Must |
| F-SRCH-02 | ゲームモード、ランク帯、募集レーン、雰囲気タグ、VC 有無で絞り込みできる | Must |
| F-SRCH-03 | フリーワード検索ができる | Should |
| F-SRCH-04 | 「自分のランク・レーンに合う募集」を優先表示するおすすめ機能 | Could |

### 4.4 応募・承認

| ID | 要件 | 優先度 |
|---|---|---|
| F-APP-01 | ログインユーザーは募集中の募集に、希望レーンと一言コメントを添えて応募できる | Must |
| F-APP-02 | 募集主は応募者のプロフィール(ランク・レーン・評価)を確認し、承認/見送りできる | Must |
| F-APP-03 | 承認された応募者は参加者となり、募集の参加者枠に表示される | Must |
| F-APP-04 | 応募者は承認前に応募を取り下げできる。参加者は参加を辞退できる | Must |
| F-APP-05 | 同一募集への重複応募、自分の募集への応募はできない | Must |
| F-APP-06 | 応募ステータスは「応募中 → 承認 / 見送り / 取り下げ / 辞退」で管理する | Must |
| F-APP-07 | 募集主が応募を一定時間放置した場合、応募者へその旨を通知する | Could |

### 4.5 マッチング成立後(Discord 集合導線)

| ID | 要件 | 優先度 |
|---|---|---|
| F-DSC-01 | 募集主は募集に Discord サーバー招待リンク(または VC の URL)を設定できる | Must |
| F-DSC-02 | 招待リンクは**承認済みの参加者にのみ**表示される(募集ページの一般公開部分には出さない) | Must |
| F-DSC-03 | 募集成立時、参加者全員に集合案内(招待リンク・開始時刻・参加者一覧)を提示する | Must |
| F-DSC-04 | 参加者一覧には各メンバーの Discord 名・Riot ID・担当レーンを表示する | Must |
| F-DSC-05 | 公式 Discord Bot が一時的な専用 VC/テキストチャンネルを自動作成する | Could |

| F-NTF-02 | 応募者へ: 承認/見送りされたとき通知する | Must |
| F-NTF-03 | 参加者へ: 開始時刻の一定時間前(例: 15 分前)にリマインドする | Should |

### 4.7 安全・健全性

| ID | 要件 | 優先度 |
|---|---|---|
| F-SAFE-01 | ユーザー・募集を通報できる(理由選択+自由記述)。通報理由には「サブ垢・スマーフの疑い」を含める | Must |
| F-SAFE-02 | ユーザーをブロックできる。ブロックした相手の募集は非表示になり、相手は自分の募集に応募できない | Must |
| F-SAFE-03 | プレイ終了後、参加者同士で評価(GG レビュー: Good / Bad +任意コメント)を送れる | Should |
| F-SAFE-04 | 評価の集計値(GG 率など)をプロフィールに表示する | Should |
| F-SAFE-05 | 無断キャンセル(承認後の無連絡不参加)を記録し、プロフィールに反映する | Could |
| F-SAFE-06 | コミュニティガイドライン・利用規約・プライバシーポリシーを掲示し、登録時に同意を取得する | Must |
| F-SAFE-07 | 運営者は通報を確認し、ユーザーの警告・凍結・募集の非公開化ができる(管理画面) | Must |
| F-SAFE-08 | NG ワードフィルタにより不適切な募集文・コメントを抑止する | Should |
| F-SAFE-09 | ランクと募集条件の不一致(API 取得ランクが募集のランク帯条件を満たさない応募)はシステムが警告または応募不可とする | Should |

## 5. 非機能要件

| ID | 分類 | 要件 |
|---|---|---|
| N-01 | 性能 | 募集一覧・検索は通常時 1 秒以内に表示する |
| N-02 | 性能 | ピーク帯(21〜25 時)の同時アクセスを想定した設計とする(MVP 目標: 同時 500 ユーザー) |
| N-03 | 可用性 | 稼働率 99.5% 以上を目標とする(個人開発規模の現実的な目標値) |
| N-04 | セキュリティ | Discord OAuth トークンはサーバー側で安全に管理し、クライアントへ露出しない |
| N-05 | セキュリティ | 全通信を HTTPS 化する。CSRF/XSS/SQL インジェクション等の基本対策を行う |
| N-06 | セキュリティ | Discord 招待リンクは承認済み参加者以外に漏れないようアクセス制御する |
| N-07 | プライバシー | 個人情報の取り扱いはプライバシーポリシーに明記し、退会時に削除する |
| N-08 | 年齢配慮 | 利用規約で 13 歳以上(Discord 規約準拠)を利用条件とする |
| N-09 | UI | スマートフォン優先のレスポンシブデザインとする(SNS からの流入が主のため) |
| N-10 | UI | 日本語 UI。ダークテーマ基調(ゲーマーの利用環境を考慮) |
| N-11 | 拡張性 | ゲームタイトルをマスタデータ化し、コード変更を最小限に新ゲームを追加できる構造とする |
| N-12 | 運用 | エラー監視・アクセスログを取得し、障害時に原因を追跡できる |
| N-13 | 外部連携 | Riot API のレートリミットとポリシーを遵守する。取得結果をキャッシュし、API 障害時もキャッシュ値(最終取得日時付き)でサービスを継続できる |
| N-14 | 外部連携 | Riot API キーは本番用(Production API Key)の承認を取得して運用する |

## 7. データモデル(概要)

```
Game 1 ──< Recruitment >── 1 User(募集主)
              │
              └──< Application >── 1 User(応募者)

User ──< Review >── User      (プレイ後の相互評価)
User ──< Report                (ユーザー/募集への通報)
User ──< Block >── User
User ──< Notification
```

### 主要エンティティ

| エンティティ | 主な属性 |
|---|---|
| **User** | id, discord_id(**unique**), discord_name, avatar_url, riot_id, riot_puuid(**unique**), rank_solo, rank_flex, rank_fetched_at(ランクは Riot API 取得値のみ), lanes[], play_hours, vc_style, bio, gg_rate, status(active/warned/suspended), created_at, deleted_at |
| **Game** | id, name, slug, modes[], rank_tiers[], lanes[](ゲームごとの役割定義。LoL なら TOP/JG/MID/ADC/SUP) |
| **Recruitment** | id, game_id, owner_id, mode, slots[](レーン×人数), rank_min, rank_max, start_at, duration, vc_required, vc_tool, tags[], comment, discord_invite_url, status(open/filled/closed/expired), created_at |
| **Application** | id, recruitment_id, applicant_id, desired_lane, comment, status(pending/approved/rejected/withdrawn/declined), created_at |
| **Review** | id, from_user_id, to_user_id, recruitment_id, rating(good/bad), comment |
| **Report** | id, reporter_id, target_type(user/recruitment), target_id, reason, detail, status |
| **Block** | id, user_id, blocked_user_id |
| **Notification** | id, user_id, type, payload, read_at, created_at |
| **SanctionRecord** | id, discord_id, type(warning/suspension), reason, created_at(退会後も Discord ID 単位で保持し、再登録時に引き継ぐ → F-UNIQ-04) |

> 拡張ポイント: ランク帯・レーン・モードは `Game` 側の定義を参照する。LoL 固有の値をコードに埋め込まないことで、フェーズ 3 のゲーム追加をマスタ登録のみで行えるようにする。

## 8. ロードマップ(MVP の線引き)

### MVP(フェーズ 1)に含める

- Discord OAuth ログイン、プロフィール(F-ACC-01〜08)

## 9. 本リポジトリのプロトタイプについて

本リポジトリには上記要件の画面イメージを確認するための**静的 HTML プロトタイプ**(バックエンドなし・サンプルデータ表示)を同梱する。

| ファイル | 対応画面 |
|---|---|
| `index.html` | S-01 トップ / 募集一覧(フィルタ動作はクライアント JS で再現) |
| `recruit-detail.html` | S-02 募集詳細(応募・参加者枠・Discord 案内のモック) |
| `recruit-new.html` | S-03 募集作成フォーム |
| `mypage.html` | S-04 マイページ(承認管理のモック) |
| `assets/style.css` / `assets/app.js` | 共通スタイル・サンプルデータ・フィルタ処理 |

応募ボタン・承認ボタン等は画面遷移イメージ確認のためのモック動作であり、データは保存されない。
